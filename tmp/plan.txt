Plan: Robust Kotlin-style Lambda Support for UnholyC Transpiler

     Context

     Add Kotlin-style trailing lambda syntax to transpiler.cpp. The goal is full robustness — not just the 4 documented examples, but all reasonable combinations of arguments, types, captures, nesting, and call-site patterns.

     ---
     Architecture: Transpiler class + Two-pass output

     The existing transpile() free function becomes a thin wrapper around a new Transpiler class. This avoids a full rewrite while adding the necessary state:

     class Transpiler {
         const std::vector<Token>& tokens;
         const std::unordered_set<std::string>& namespaces;
         std::ostringstream out;
         std::ostringstream preamble;
         // ... (all state below)
     public:
         std::string run();  // replaces transpile()
     };

     std::string transpile(tokens, namespaces) { return Transpiler(tokens, namespaces).run(); }

     Two passes:
     - Pass 1 (main loop): emit tokens to out, emit generated lambda functions to preamble, record PatchSites for callee signature patching
     - After pass 1: validate capture set consistency, apply patches to out (reverse-sorted by offset), return preamble.str() + out.str()

     ---
     New Data Structures

     struct ScopeVar      { std::string name, cType; };
     struct LambdaReg     { std::string funcName, lambdaParamName;
                            std::vector<std::string> argCTypes, retCType; };
     struct PatchSite     { size_t outputByteOffset; std::string calleeName; };
     struct CallSiteCap   { std::string calleeName; int sourceLine;
                            std::vector<std::pair<std::string,std::string>> captures; }; // (name,cType)
     struct CallFrame     { std::string calleeName; int parenDepthAtOpen; };

     ---
     New Helper Functions

     parseCType(i) → string

     Parses one C type from the token stream. Handles:
     - TYPE_MAP identifiers: I32 → int
     - Namespace-dot types: MyNS.Type → MyNS::Type
     - Pointer/ref suffixes (*, &): consumed as TK::OTHER
     - unused prefix: appends __attribute__((unused))
     - Unknown identifiers: pass through unchanged

     parseLambdaDecl(i) → string

     Called when IDENT("lambda") is seen at parenDepth > 0.
     Consumes: lambda NAME ( types... ) -> rettype
     Emits: retCType (*NAME)(argCTypes...)
     Side-effect: registers the lambda in lambdaRegistry.

     extractBody(i) → {headerParams, bodyTokens}

     Called when LBRACE is identified as a trailing lambda body.
     - Parses optional (param, ...) -> header
     - Extracts body tokens with brace-depth counter (always correct because lexer makes strings/comments/chars atomic tokens — {/} inside them are never TK::LBRACE/TK::RBRACE)
     - Returns header param names + body token list

     detectCaptures(headerParams, bodyTokens) → vector<(name,cType)>

     - Builds headerSet from declared param names
     - Scans body tokens for TK::IDENT not in headerSet, not a keyword/type/namespace
     - For each such identifier, checks scopeVars (includes function params)
     - Returns captures in order of first appearance

     generateLambdaName(bodyTokens) → string

     - FNV-1a hash of concatenated body token values → 8 hex digits → __lambda_XXXXXXXX
     - Counter suffix _2, _3... on collision (same hash seen twice in the same file)

     emitLambdaFunction(name, retCType, argCTypes, headerParams, bodyTokens, captures)

     - Writes to preamble
     - Signature: retCType name(arg0Type param0, ..., capType capName, ...)
     - Body: body tokens processed through transpileTokens() (same TYPE_MAP + namespace logic, may recursively emit nested lambdas into preamble)

     applyPatches(outStr)

     - Sorts patchSites by outputByteOffset descending
     - For each site, inserts , capType capName into outStr at the recorded offset
     - Also patches the corresponding lambda function-pointer type inside the same param list

     ---
     Scope Tracking (inside run() main loop)

     - On entering a top-level function body (LBRACE at brace depth 0→1): clear scopeVars, parse and record function's own parameters into scopeVars, set inFunctionBody = true
     - On each LBRACE inside body: scopeDepth++
     - On each RBRACE inside body: scopeDepth--; if back to 0 then inFunctionBody = false, clear scopeVars
     - Variable declaration detection: pattern <type-token(s)> IDENT followed by =, ;, or , while in function body → record ScopeVar{ident, resolvedCType}
     - Variables declared anywhere in the function up to the lambda call site are capture candidates (flat scope; conservative but always safe)

     ---
     Call Site Detection (inside run() main loop)

     With parens: IDENT LPAREN ... RPAREN followed by LBRACE
     - Use callStack (vector of CallFrame) to track the current call's name
     - When RPAREN brings parenDepth to the frame's entry level, peek ahead for LBRACE
     - If found: trigger lambda extraction, emit , __lambda_XXX (and capture args) before the ), then skip the brace body

     Without parens: IDENT directly followed by LBRACE
     - After emitting an IDENT, peek for LBRACE (skipping whitespace)
     - Only trigger if the IDENT is registered as a function taking a no-arg lambda (check lambdaRegistry)
     - Emit ident(__lambda_XXX) and skip the brace body

     Exclusions (never trigger):
     - Control-flow keywords: if, for, while, do, switch, else
     - When parenDepth > 0 at the moment LBRACE is detected → emit error: Trailing lambda cannot appear as a function argument or condition
     - Two consecutive { } blocks after one call → emit error: Only one trailing lambda block per call site

     ---
     Lambda Declaration -> Token Handling

     -> is lexed as two TK::OTHER tokens: - then >. The parser only interprets this as the lambda return-type separator when inside an active parseLambdaDecl() invocation. Inside lambda bodies, - and > are emitted as-is.

     ---
     Callee Signature Patching

     Same-file callee (supported):
     1. When parseLambdaDecl() emits the C function-pointer syntax, record a PatchSite at the byte offset of the closing ) of the callee's parameter list in out
     2. After the main loop, for each CallSiteCapture, validate consistency: all call sites of the same callee must capture the same set of variable names (types can differ only if names differ) — otherwise emit error: Incompatible capture sets across call sites for
     '<callee>'
     3. Apply patches via applyPatches(): insert , capType capName for each capture, also patch the lambda function-pointer type to add the extra param types

     Forward declarations (same file, both decl + definition):
     Multiple PatchSite entries for the same callee name — all are patched.

     Cross-file callee (not supported in V1):
     If the call site has captures but the callee's PatchSite was never recorded (callee not found in this file): emit warning: Cannot auto-patch '<callee>' (defined in another file). Add capture parameters manually.

     Idempotency:
     Before inserting a capture param, check if that param name already exists in the callee's recorded param token list. Skip if already present.

     ---
     Nested Lambdas

     A lambda body may itself call a function with a trailing lambda. The transpileTokens() function used to emit the body is the same recursive path as the main loop, sharing the same preamble buffer. Nested lambda functions are emitted into preamble before the outer
     lambda — resulting in correct definition order in the output.

     ---
     Implementation Layers (sequential, each testable)

     1. Infrastructure: Transpiler class refactor, parseCType(), generateLambdaName(), preamble buffer, applyPatches() stub
     2. Lambda declaration parsing: parseLambdaDecl(), lambdaRegistry, PatchSite recording
     3. Scope tracking: function entry/exit, param tracking, variable declaration detection
     4. Body extraction + generation: extractBody(), detectCaptures(), emitLambdaFunction(), call-site detection (callStack), body-skip logic
     5. Callee patching: CallSiteCapture collection, consistency validation, applyPatches() implementation
     6. Edge cases + diagnostics: all error/warning paths, nested lambdas, hash collision, unused in lambda params

     ---
     Critical Files

     - transpiler.cpp — only file modified; ~400–500 lines added
     - lambda.txt — primary test oracle (all 4 documented cases)

     Verification

     1. Build: g++ -std=c++17 transpiler.cpp -o transpiler
     2. Create .uhc test files from each section of lambda.txt, run transpiler, diff against expected output
     3. Test edge cases: empty body, nested braces, string with { inside body, nested lambda, multiple captures, cross-file warning
     4. Run transpiler on uhcstd/ to verify existing behavior (namespaces, unused, types) is unchanged
